\section{Introduction}
A \squeeze{} is a paper cast taken from an inscribed stone surface. It is not an ink image. It is a
shallow relief surface, photographed under raking light, where a letter stroke can be clear in one
illumination direction and nearly absent in the other. Surface wear, paper grain, broken relief, and
the geometry of the light all decide what evidence is visible.

The deployed recognizer treats this as a physical multi-view reading problem. The visual backbone is
\model{}, a dual-light TrOCR line-recognizer operating on horizontal crops containing up to 4 characters (referred to as tile4 crops) that packs both raking-light scans into one RGB tensor.
The production transcript is then selected by a \charpost{} correction layer: a per-character posterior
lattice, train-only character language models, and a learned ridge row ranker. In short, the line model
supplies strong visual evidence; the lattice layer turns that evidence into letter-by-letter candidate
rows and chooses the row that is visually and linguistically most plausible.

The active artifact tree is rooted at \texttt{data/}. It contains one all-train line recognizer,
five fold-excluded line recognizers, fold-clean character-posterior classifiers, character language
models, the OOF ridge ranker, and decode artifacts. The cache-guarded notebook
\texttt{greek\_squeezes.ipynb} validates or rebuilds the same tree locally, while \texttt{modal.py}
builds and applies it on Modal.

The most recent external production evaluation was run on
\texttt{papadimas/trogs-26-test-images}, split \texttt{test}. This separate dataset has 50 squeezes,
100 images, 605 scored rows, and 7,919 scored characters. The frozen production pipeline scored
\cer{} \textbf{0.0822} (651 character errors). The run summary is staged under
\texttt{data/prod\_runs/trogs26-test/summary.json} in the project Hugging Face bucket.

\section{Data and Evaluation Hygiene}
The internal TROGS development data keeps both scans of a physical squeeze in the same split or fold.
This matters because \texttt{Rotation1} and \texttt{Rotation2} are two photographs of the same
inscription, not independent examples. The line recognizers are trained from official train
material only; the five OOF recognizers each exclude their held-out fold. Clean OOF means the held-out
fold is excluded from three places:

\begin{enumerate}
  \item the fold TrOCR checkpoint used to initialize visual evidence,
  \item the fold character-posterior classifier,
  \item the fold train-only character n-gram language models.
\end{enumerate}

Validation and internal test outputs are report-only once the recipe is frozen. External production
sets, including \texttt{trogs26-test}, are not used for training, checkpoint selection, ranker fitting,
or hyper-parameter choice.

\section{Dual-Light Line Recognition}
\subsection{Physical motivation}
Raking light is useful precisely because it is directional. A groove or ridge casts a shadow when the
light crosses it, but can flatten into the paper texture when the light runs along it. The two scans are
therefore complementary: one view may reveal a vertical stroke while the other reveals the crossbar or
the opposite ridge wall. Decoding the scans separately asks the recognizer to choose a photograph before
it has read the text. Dual-light packing instead gives the encoder both photographs at once.

\subsection{Input representation}
The line recognizer is the \model{} recipe: tile4 crops (horizontal crops containing up to 4 characters), a
\texttt{microsoft/trocr-large-printed} encoder-decoder, and dual-light RGB packing. The paired crops are
phase-registered and packed into the three channels that TrOCR already expects:

\begin{center}
\small
\begin{tabular}{@{}ll@{}}
\toprule
\textbf{Channel} & \textbf{Content} \\
\midrule
R & \texttt{Rotation1} crop \\
G & \texttt{Rotation2} crop, phase-registered to \texttt{Rotation1} \\
B & pixelwise \texttt{min(R,G)}, a dark-stroke union image \\
\bottomrule
\end{tabular}
\end{center}

The third channel is the key visual trick. It is not a natural color image; it is a dark-stroke union.
If either lighting direction makes a groove dark, \texttt{min(R,G)} keeps that evidence dark. Training
also uses random R/G channel swap and mono dropout 0.15, so the model cannot rely on a fixed lighting
order or a single view. At evaluation time those stochastic augmentations are disabled, while phase
registration and \texttt{DUAL\_MODE=min} remain enabled.

\subsection{Training recipe}
The all-train line checkpoint at \texttt{data/line\_recognizer/all\_train} uses:

\begin{itemize}
  \item base model \texttt{microsoft/trocr-large-printed};
  \item processor base \texttt{microsoft/trocr-base-printed};
  \item real crops only, no synthetic data;
  \item tile4 crops of up to 4 characters (\texttt{lmc=4});
  \item 6000 optimizer steps, batch 96, gradient accumulation 1;
  \item learning rate \(2\times 10^{-5}\), gradient checkpointing, validation gate \(\cer{} < 0.20\);
  \item dual-light phase registration with max shift 0.15.
\end{itemize}

The fold checkpoints under \texttt{data/line\_recognizer/oof\_fold0} through
\texttt{oof\_fold4} use the same recipe, but each excludes the corresponding OOF fold.

\section{Lattice Decoder and Ranker}
The deployed correction layer turns the line recognizers into a per-character posterior pipeline. It
extracts tile1 crops (single-character crops) from the letter boxes, trains a classifier over the explicit
\charpost{} alphabet, and caches logits for each row position. This converts a row into a lattice:
every position has a probability distribution over possible proxy characters. The row decoder can then
separate two failure modes that are hard to distinguish in a single line string: the correct row may be
absent from the visual lattice, or it may be present but ranked below a visually similar alternative.

For OOF fitting, each fold has its own:

\begin{itemize}
  \item fold-excluded \charpost{} classifier initialized from the fold-excluded line recognizer;
  \item held-out fold logits;
  \item train-only character language models of orders 6, 10, and 12.
\end{itemize}

Candidate rows are proposed by beam search over the visual lattice and over lattice+LM scores. The
current candidate search uses proposal weights \(0,0.25,0.5,0.75,1.0\), beam 256, and char-top-k 8.
The learned ranker is a ridge model over:

\begin{center}
\small
\texttt{visual\_sum, mean\_char\_logprob, length, length\_delta, lm6\_sum, lm10\_sum, lm12\_sum}.
\end{center}

The \texttt{length\_delta} feature follows the active
\texttt{lattice\_length\_delta\_v2} schema: candidate length minus lattice/source length, not candidate
length minus ground-truth length. After OOF fitting, the all-train \charpost{} classifier and
train-only LMs produce validation, internal test, and production decodes.

\section{Deployment and Reproducibility}
The active local notebook is \texttt{greek\_squeezes.ipynb}. It is cache-guarded: if artifacts are
already present under \texttt{data/}, cells validate and skip them; if artifacts are missing and
\texttt{RUN\_TRAINING\_IF\_MISSING=True}, the same runtime modules can rebuild them. On a normal laptop
the preferred path is to sync the Modal/Hugging Face artifact tree first, because training the full
\charpost{} stack locally is GPU-heavy.

The Modal entry point is \texttt{modal.py}. The typical sequence is:

\begin{enumerate}
  \raggedright
  \item \texttt{uvx modal run modal.py -{}-prepare}
  \item \texttt{uvx modal run modal.py -{}-run -{}-upload}
  \item \texttt{uvx modal run modal.py -{}-prod -{}-prod-source papadimas/trogs-26-test-images -{}-prod-source-split test -{}-prod-name trogs26-test}
  \item \texttt{uvx modal run modal.py -{}-upload-prod-result-only -{}-prod-name trogs26-test}
\end{enumerate}

Reusable artifacts are uploaded from \texttt{/root/work/data} to the project Hugging Face bucket,
\texttt{hf://buckets/papadimas/greek\_squeezes}. Production runs are staged under
\texttt{data/prod\_runs/<slug>/}. Raw images and Hugging Face caches are deliberately excluded from
the reusable artifact upload.

\section{Conclusion}
To read both raking-light views in a single forward pass, the system uses a registered RGB packing with a \texttt{min(R,G)} union channel. The deployed \charpost{} layer turns the line model into a clean OOF-trained character lattice and ridge row ranker. The resulting production pipeline is reproducible through \texttt{greek\_squeezes.ipynb}, cacheable under \texttt{data/}, runnable on Modal, and scored on a separate TROGS-26 test-image dataset at \cer{} \textbf{0.0822}.
